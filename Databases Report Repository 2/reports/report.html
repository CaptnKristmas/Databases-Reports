This assignments goal is to pick three problems and create Chen and Crow’s foot diagrams. These diagrams represent Entity Relation models that visually describe the attributes and relationships between the separate entitities.

Entities are larger concepts. These larger concepts can then be broken down into several attributes that are important to discerning the entity. Each entity has a primary key, this is an attribute that identifies each individual entity from another. Say you need to manage students and their courses, a situation I model below, Students and Courses would be entities. There are many students in a school and many courses. In order to discern different students and different courses they each have primary key attributes, like studentID or courseID. Relationships are made between them based on necessities for the system. Such as each student can have many classes and classes can have many students, this for example is a many to many relationship.

Entities are squares, attributes are ovals (primary keys are usually underlined or labled), and diamonds are relationships in Chen Diagrams. The numbers on the lines are the cardinality or how much of each must be inlcuded and how many can be included in that order. N or M represent many and are used interchangably. If you see both, it represents different many’s.

In Crows Foot Diagrams tables titles are entities, attributes are listed under the table and the primary key is listed on the table in a separate column. There is also a separate column for the type of attribute, int or string etc. Their relationship is showed to the side. A trident represents many and two lines represent at least one.

ER Modeling Exercises

School Course Enrollment System

Imagine a school that needs to manage Courses, Students, and enrollments. Each Course has a Course ID, title, and credits. Students have a Student ID, name, and enrollment date. Enrollments have an enrollment ID, enrollment date, and grade. A Student can enroll in multiple Courses, and each Course can have multiple Students. Each enrollment is associated with a single Student enrolling in a single Course.

Conceptual design in Chen Notation

The diagram depicts a many to many relationship between Students and Courses resolved via the associative entity Enrollment. Courses and Students each have a one to many relationship with Enrollments. Students and Courses must each refer to exactly one Enrollment. Each of these entities has their own attributes. CourseID being the primary key for Course, EnrollementID being the primary key for Enrollements and StudentID being the primary key for Students.

Conceptual design in Crows foot notation

Entities are Courses, Students and Enrollments. Students has attributes: int courseID (the primary key), string title, and string credits. Courses has attributes: int courseID (the primary key), string name, and int enrollmentDate. Enrollments has attributes: int enrollmentID (the primary key), int enrollmentDate, int grade. Student is connected via a one to many takes relationship to Enrollment and Course is connected via a one to many has relationship to Enrollement.

Design assumptions, considerations and tradeoffs

The first assumption I had was to make Students have a many to many relationship with Courses and then a one to one relationship with Enrollments. I then questioned how this would properly connect to Courses to track the information leading me to believe a ternary relationship would be best. I remembered it is best to avoid this and decided using Enrollments as the associative identity would be best.

Relation sets

The following are the initial relation sets for the system:

  • Course( courseID, title, credits )
  • Students( studentID, name, enrollmentDate )
  • Enrollment( enrollmentID, enrollmentDate, Grade )

Healthcare Appointment Management System

Consider a healthcare clinic that wants to manage patients, doctors, and appointments. Each patient has a patient ID, name, date of birth, and contact information. Doctors have a doctor ID, name, specialty, and contact number. Appointments have an appointment ID, date, time, and status. A patient can schedule multiple appointments, and each appointment is with a single doctor. A doctor can have multiple appointments, but each appointment is associated with only one patient.

Conceptual design in Chen Notation

The diagram depicts a many to one relationship between Doctors and Appointments and many to one relationship between Patients and Appointments. Each Appointment must have exactly one Doctor and one Patient. Each of these entities has several attributes with doctorID being the primary key for Doctors, appointmentID being the primary key for Appointments, and patientID being the primary key for Patients.

Conceptual design in Crows foot notation

The entities are Patients, Doctors and Appointments. Patients has attributes: int patientID (the primary key), int dob, string name, and string contactInformation. Doctors has attributes: int doctorID (the primary key), string name, string specialty, int contactNumber. Appointments has attributes: int appointmentID (the primary key), int time, int date, and int status. Patient is connected via a one to many makes relationship with Appointment and Doctor is conncted via a one to many relationship with Appointment.

Design assumptions, considerations and tradeoffs

This was less difficult to process for me then the first one. I could see the relationships that needed to be made pretty easily. The main consideration was how to treat certain attributes. Is status an int or a string? What about dob and contactInformation? I decided that status would be best as an int to represent several different states. Dob would be easiest to deal with as an int so I chose that. Contact Information could be an email as well so I chose string.

Relation sets

The following are the initial relation sets for the system:

  • Patient( patientID, dob, name, contactInformation )
  • Appointment( appointmentID, status, time, date )
  • Doctor( doctorID, name, contactNumber, specialty )

Restaurant Reservation System

Envision a restaurant that wants to manage reservations, customers, and tables. Each reservation has a reservation ID, date, and time. Customers have a customer ID, name, and contact number. Tables have a table number, seating capacity, and location. A customer can make multiple reservations, and each reservation is for a single table. Each table can have multiple reservations over time, but each reservation is associated with only one customer

Conceptual design in Chen Notation

The diagram depicts a many to one relationship between Table and Reservation and many to one relationship between Customer and Reservation. Each Reservation must have exactly one Table and one Customer. Each of these entities has several attributes with tableNumber being the primary key for Table, reserveationID being the primary key for Appointments, and customerID being the primary key for Customer.

Conceptual design in Crows foot notation

The entities are Tables, Reservations and Customers. Tables has attributes: int tableNumber (the primary key), int seatingCapacity, and string location. Reservations has attributes: int reservationID (the primary key), int time, and int date. Customers has attributes: int customerID (the primary key), string name, and int contactNumber. Customer is connected via a one to many makes relationship with Reservation and Table is conncted via a one to many relationship with Reservation.

Design assumptions, considerations and tradeoffs

Similar to the last one the relationships were pretty easy to figure out. This time the consideration was as to what was the primary key for table. Should it be location or tableNumber? I ended up deciding the best choice was tableNumber due to location possibly being similar.

Relation sets

The following are the initial relation sets for the system:

  • table( tableNumber, location, seatingCapacity )
  • reservation( reservationID, date, time )
  • customer( customerID, contactNumber, name )
